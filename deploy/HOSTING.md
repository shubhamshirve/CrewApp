# Dual-App Hosting Guide — Photoo + JVSapp on one VPS

> **VPS:** `45.196.196.114` (root)
> **Photoo:** [`https://app.photoo.in`](https://app.photoo.in) — this repo (`CrewApp`)
> **JVSapp:** [`https://app.mmpf.in`](https://app.mmpf.in) — separate repo
>
> Both apps share the same VPS for ~2–3 months, then move to dedicated servers.
> Nothing in either app's source code is edited — co-hosting is achieved through
> a **host-level Caddy** and a single non-destructive `docker-compose.override.yml`.

---

## 1. The mental model

The VPS has exactly **one process** owning ports `80` and `443`: the **host Caddy**, installed via apt at `/etc/caddy/Caddyfile`. Everything else runs in Docker on `127.0.0.1`-bound high ports that only the host can reach.

```
                              Internet  (HTTPS, port 443)
                                       │
                                       ▼
                          ┌──────────────────────────────┐
                          │   Host Caddy   (apt service) │   /etc/caddy/Caddyfile
                          │   listens on :80 + :443      │   - auto Let's Encrypt
                          │   terminates TLS for both    │   - HTTP/2 + HTTP/3
                          │   apps; selects upstream by  │
                          │   the SNI host header.       │
                          └──────────┬───────────────────┘
                                     │
              ┌──────────────────────┴──────────────────────┐
              │                                             │
              ▼  127.0.0.1:3000                             ▼  127.0.0.1:8080
   ┌────────────────────────────┐                ┌────────────────────────────────┐
   │ /opt/photoo                │                │ /opt/jvsapp                    │
   │ docker compose             │                │ docker compose                 │
   │                            │                │                                │
   │ ┌────────────────────────┐ │                │ ┌────────────────────────────┐ │
   │ │ photoo-frontend (Caddy)│ │                │ │ jvsapp caddy service       │ │
   │ │   /api/* → backend     │ │                │ │   (serves React + /api)    │ │
   │ │   /*     → React SPA   │ │                │ └────────────────────────────┘ │
   │ └─────────┬──────────────┘ │                │ + backend + mongo containers   │
   │           │ Docker DNS     │                │                                │
   │ ┌─────────▼──────────────┐ │                │ docker-compose.override.yml    │
   │ │ photoo-backend         │ │                │  ← AUTO-GENERATED, re-maps     │
   │ │  (FastAPI 8001)        │ │                │    the caddy service from      │
   │ └─────────┬──────────────┘ │                │    host 80/443 to 8080/8443.   │
   │           │                │                │    The repo's own              │
   │ ┌─────────▼──────────────┐ │                │    docker-compose.yml is       │
   │ │ photoo-mongodb (4.4)   │ │                │    NEVER edited.               │
   │ └────────────────────────┘ │                │                                │
   │                            │                │                                │
   │ Network: photoo-network    │                │ Network: (jvsapp's own)        │
   └────────────────────────────┘                └────────────────────────────────┘
```

### Port allocation (host)

| Port | Owner | Purpose |
|------|-------|---------|
| `22` | sshd | SSH |
| `80` | **host Caddy** | HTTP (redirects to HTTPS) |
| `443` | **host Caddy** | HTTPS for both apps |
| `127.0.0.1:3000` | photoo-frontend (Docker) | Photoo SPA + reverse proxy to backend |
| `127.0.0.1:8080` | jvsapp caddy (Docker) | JVSapp HTTP (forwarded by host Caddy) |
| `127.0.0.1:8443` | jvsapp caddy (Docker) | JVSapp internal HTTPS (unused — host Caddy does TLS) |
| nothing | photoo-backend | Docker-internal only, never published |
| nothing | photoo-mongodb | Docker-internal only, never published |

**No port collision is possible**, because both Docker compose stacks bind exclusively to `127.0.0.1` and use distinct high ports.

### Why a `docker-compose.override.yml` (not a fork of JVSapp)

JVSapp's repo defines `caddy: ports: ["80:80", "443:443"]` — which would steal the host ports we need for shared Caddy. The script generates this file at `/opt/jvsapp/docker-compose.override.yml`:

```yaml
# AUTO-GENERATED by Photoo deploy.sh
services:
  caddy:
    ports: !override
      - "127.0.0.1:8080:80"
      - "127.0.0.1:8443:443"
```

Docker Compose auto-merges `docker-compose.override.yml` whenever someone runs `docker compose up` in `/opt/jvsapp`. The `!override` tag (Compose v2.24+) **replaces** the ports list instead of appending to it.

**To move JVSapp to its own server later:**

```bash
rm /opt/jvsapp/docker-compose.override.yml
# JVSapp's own docker-compose.yml takes over 80/443 again, untouched.
```

## 2. First-time deployment (fresh VPS)

```bash
ssh root@45.196.196.114
curl -fsSL https://raw.githubusercontent.com/shubhamshirve/CrewApp/main/deploy/deploy.sh -o /tmp/deploy.sh
bash /tmp/deploy.sh
```

The script is **idempotent and self-healing** — re-running it after a failure won't break anything.

**Before you run, confirm DNS:**

```bash
dig +short app.photoo.in   # must return 45.196.196.114
dig +short app.mmpf.in     # must return 45.196.196.114
```

If DNS isn't propagated, Caddy will still start, but Let's Encrypt will fail to issue certs. Re-run the script after DNS resolves correctly — Caddy automatically retries every 5 minutes anyway.

**The 8 steps the script performs:**

1. Installs host Caddy via apt (if missing).
2. Detects + migrates any legacy `crewbook_*` Docker volumes → `photoo_*` (no-op on a fresh VPS).
3. Clones JVSapp at `/opt/jvsapp` (or pulls if already cloned).
4. Generates `/opt/jvsapp/docker-compose.override.yml` to remap ports 80/443 → 8080/8443.
5. Clones Photoo at `/opt/photoo` and generates `backend/.env` with a fresh 64-char `JWT_SECRET` + 32-char `ADMIN_SEED_SECRET`.
6. `docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build` for both stacks.
7. Writes `/etc/caddy/Caddyfile` with both domains and Let's Encrypt enabled.
8. Runs HTTP probes on `127.0.0.1:3000` + `127.0.0.1:8080` to confirm reachability.

## 3. Day-2 operations cheatsheet

### Update Photoo to latest commit
```bash
cd /opt/photoo
git pull
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build
```
Thanks to the **BuildKit pnpm cache mount**, the frontend rebuild step takes ~30 s when `pnpm-lock.yaml` hasn't changed (down from ~18 minutes on the first build).

### Update JVSapp to latest commit
```bash
cd /opt/jvsapp
git pull
docker compose up -d --build
```
The override file persists across pulls — JVSapp's repo doesn't even know it exists.

### Restart only Photoo's backend (after editing `backend/.env`)
```bash
cd /opt/photoo
docker compose -f docker-compose.yml -f docker-compose.prod.yml restart backend
```

### Reload host Caddy after editing `/etc/caddy/Caddyfile`
```bash
caddy validate --config /etc/caddy/Caddyfile
systemctl reload caddy
```
`reload` is zero-downtime; certificates are preserved.

### Tail logs

```bash
# Photoo (all services)
cd /opt/photoo && docker compose -f docker-compose.yml -f docker-compose.prod.yml logs -f

# JVSapp (all services)
cd /opt/jvsapp && docker compose logs -f

# Host Caddy (TLS issuance, request logs)
journalctl -u caddy -f
```

### Seed Photoo demo data (idempotent)
```bash
cd /opt/photoo
docker compose -f docker-compose.yml -f docker-compose.prod.yml \
    exec backend python /app/scripts/seed_data.py
```

## 4. Adding a third app later

Append a new block to `/etc/caddy/Caddyfile`:

```caddy
newapp.example.com {
    encode gzip zstd
    reverse_proxy 127.0.0.1:9000 {
        header_up X-Forwarded-For {remote_host}
        header_up X-Forwarded-Proto {scheme}
    }
}
```

Then run the new app's Docker stack with all host ports re-mapped to `127.0.0.1:9000` (or any free high port). `systemctl reload caddy`. Done.

## 5. Moving one app to its own server

### Move JVSapp to a dedicated VPS

```bash
# On the shared VPS:
cd /opt/jvsapp && docker compose down
rm docker-compose.override.yml                # restore 80/443 defaults
# Remove the app.mmpf.in block from /etc/caddy/Caddyfile, then:
systemctl reload caddy

# On the new VPS (no Photoo there):
git clone https://github.com/shubhamshirve/JVSapp.git /opt/jvsapp
cd /opt/jvsapp && docker compose up -d        # works as the repo intends
# Point app.mmpf.in DNS at the new IP.
```

### Move Photoo to a dedicated VPS

```bash
# On the new VPS:
git clone https://github.com/shubhamshirve/CrewApp.git /opt/photoo
cd /opt/photoo

# Option A — keep the host-Caddy pattern (recommended for HTTPS auto-management)
cp deploy/caddy-vps.conf /etc/caddy/Caddyfile
# Remove the app.mmpf.in block since JVSapp is no longer co-hosted
systemctl reload caddy
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build

# Option B — let the frontend container's internal Caddy own 80/443
# In docker-compose.prod.yml, change "127.0.0.1:3000:80" → "0.0.0.0:80:80"
# Add "0.0.0.0:443:443" + mount a Caddyfile that does TLS for app.photoo.in.
# No host Caddy needed.
```

## 6. Backups (mandatory before any DB migration)

```bash
# Dump Photoo's MongoDB (running container)
docker exec photoo-mongodb mongodump --archive --gzip > photoo-backup-$(date +%F).gz

# Snapshot all named volumes
for v in photoo_mongodb_data photoo_uploads_data; do
    docker run --rm -v "$v:/data" -v "$PWD:/backup" alpine:3 \
        tar czf "/backup/$v-$(date +%F).tgz" -C /data .
done
```

Restore is the reverse — `mongorestore --archive --gzip < photoo-backup-YYYY-MM-DD.gz`.

## 7. Troubleshooting

| Symptom | Probable cause | Fix |
|---|---|---|
| Caddy fails to bind `:80` / `:443` | Old JVSapp container still listening | Confirm `/opt/jvsapp/docker-compose.override.yml` exists, then `cd /opt/jvsapp && docker compose up -d --force-recreate` |
| TLS cert not issued | DNS not propagated | `dig +short app.photoo.in app.mmpf.in` must both return `45.196.196.114` |
| `502 Bad Gateway` from Caddy | The Docker container is down/unhealthy | `docker ps` — restart the failing service |
| Photoo loads, `/api/*` returns 502 | photoo-backend down or unhealthy | `docker compose -f docker-compose.yml -f docker-compose.prod.yml logs backend` |
| CORS errors in browser console | `CORS_ORIGINS` mismatch | Edit `/opt/photoo/backend/.env` so `CORS_ORIGINS=https://app.photoo.in`, then restart backend |
| `unknown tag !override` | Docker Compose < 2.24 | Upgrade Docker (`apt install docker-ce-cli` from Docker's repo) or hand-edit JVSapp's `docker-compose.yml` to bind `8080:80` directly |
| Need to abort + revert everything | — | `rm /opt/jvsapp/docker-compose.override.yml && cd /opt/jvsapp && docker compose up -d`; restore old Caddyfile from `/etc/caddy/Caddyfile.bak.*` |

## 8. Security posture (already enforced by the stack)

- ☑ MongoDB never exposed on the host network
- ☑ Photoo backend never exposed on the host network (`ports: []` in `docker-compose.prod.yml`)
- ☑ Photoo frontend bound to `127.0.0.1:3000` only — host Caddy is the sole public entry
- ☑ JVSapp containers bound to `127.0.0.1:8080`/`8443` only — host Caddy is the sole public entry
- ☑ `JWT_SECRET` is a fresh 64-char hex generated per-VPS by the deploy script
- ☑ Admin seed endpoint gated by `ADMIN_SEED_SECRET` header
- ☑ CORS restricted to `https://app.photoo.in`
- ☐ **You should run on the VPS** once: `ufw allow 22,80,443/tcp && ufw enable`
